                  Technische beschrijving van het beheerportaal

                  1Opbouw van het beheerportaal

                  Het beheerportaal is opgebouwd op basis van tegels.

                  Op de server zien we deze structuur terug in de bestandsopbouw.

                  1.1Opbouw van een tegel

                  Elke tegel wordt gedefinieerd door de bestanden in de map. Zo staan in de map ‘Algemeen’ de volgende bestanden:

                  • index.menu-item.json

                  • word-to-markdown.workflow.json

                  • word-to-publicatie.workflow.json

                  • word-to-werkversie.workflow.json

                  De json-bestanden met ‘menu-item’ bevatten de opbouw van elke tegel, bestanden met ‘workflow’ bevatten de opbouw van de workflow.

                  1.1.1Opbouw van een menu-item

                  Een menu-item gaat over de tegel waarin het bestand staat. In ons voorbeeld is dat tegel ‘Algemeen’. De opbouw is als volgt:

                  {

                  "label": "Algemeen",

                  "subtitle": "Algemene workflows voor het beheren en publiceren van documenten.",

                  "imageFilename": "Tile algemeen.png"

                  }

                  1.1.2Opzet van de workflows

                  De json-bestanden met ‘workflow’ genereren een tegel en een workflow-pagina.

                  Met de volgende informatie uit het workflow-bestand wordt in ons voorbeeld de tegel ‘Werkversie’ opgebouwd:

                  {

                  "name": "Werkversie",

                  "shortDescription": "Deze workflow transformeert een Word-bestand naar een ReSpec-werkversie.",

                  "imageFilename": "Thumbnail Sociaal Domein PNG-01.png",

                  …

                  1.2Mappenstructuur op de server

                  De mappenstructuur is in grote lijnen:

                  • ‘beheerportaal-api-files’ bevat alle bestanden die de api nodig heeft.

                  • ‘config’ bevat alle bestanden die de workflow en onderliggende stappen vastleggen.

                  • ‘cmd-step’ bevat alle gekoppelde command-stappen.

                  • ‘xpath-step’ bevat alle gekoppelde xpath-stappen.

                  • ‘workflow’ bevat alle vastgelegde workflows. De opbouw is analoog aan de opbouw van de schermen in het beheerportaal.

                  • ‘workflow-files’ bevat alle bestanden die door een workflow worden gegenereerd. Op deze map moet nog een aanpassing komen zodat na een bepaalde periode de gegenereerde bestanden worden gewist.

                  • ‘beheerportaal-repos’ bevat alle repositories. De workflows gebruiken deze bestanden voor de uitvoering. Alle gebruikte libraries en eventuele licenties moeten in de repository meegeleverd zijn.

                  2Opbouw van een workflow

                  Alle workflows volgen hetzelfde principe: ‘Je hebt een input, daar doe je wat mee en dan komt er een output uit’. De workflow-pagina is volgens datzelfde principe opgebouwd: ‘upload een input, voer de workflow uit, download de output’.

                  Het is echter niet zo dat je altijd een input of een output hoeft te hebben. Dit leg je vast in een workflow-bestand. Hierin leg je vast of er een input moet zijn, wat voor type bestand dit is, de stappen van de workflow, eventueel benodigde variabelen enzovoorts.

                  Hello World!

                  Een eenvoudig voorbeeld van een workflow is ‘Hello World’.

                  {

                  "name": "Hello World!",

                  "shortDescription": "Een simpel voorbeeld",

                  "description": [

                  "De output zou in tekst: "Hello World!" moeten zijn."

                  ],

                  "imageFilename": "Thumbnail Standaardisatie PNG-01.png",

                  "private": {

                  "comments": [],

                  "steps": [

                  {

                  "type": "cmd",

                  "command": "echo \"Hello World!\""

                  }

                  ]

                  }

                  }

                  De workflow bevat informatie voor de bovenliggende tegel en de workflow-pagina.

                  Tabel 1Eigenschappen en onderdelen van een workflow

                  Eigenschap, onderdeel

                  Omschrijving

                  name

                  Een leesbare naam van de workflow. Deze wordt gebruikt voor workflow-titel, foutmeldingen, enzovoorts. Als dit niet ingevuld is, wordt de bestandsnaam gebruikt.

                  shortDescription

                  Een beschrijving van wat de workflow doet. Deze wordt op de tegel van de workflow weergegeven.

                  description

                  Een beschrijving van wat de workflow doet. Deze wordt zichtbaar op de pagina van de workflow. Deze beschrijving kan uitgespreid worden over meerdere regels door het niet als string maar als array van strings op te geven.

                  imageFilename

                  Een van de bestandnamen van de afbeeldingen in de media-folder. Deze wordt gebruikt in de tegel van de workflow.

                  repo_name

                  Alle benodigde bestanden voor het uitvoeren van een workflow worden op de server opgeslagen als een repository. Eigenschap repo_name bevat de naam van deze repository.

                  repo_url

                  Een link naar de GitHub-repository die bij deze workflow hoort. Deze wordt alleen gebruikt om de gebruiker naar de informatie over de workflow te sturen.

                  input

                  Beschrijft de vereiste input voor de workflow. Laat je deze eigenschap weg, dan is er op de workflow-pagina geen mogelijkheid om bestanden te uploaden.

                  Als input zijn er de volgende opties:

                  • ‘files’ om vast te leggen hoeveel en welk type bestanden benodigd zijn. Onderliggende eigenschappen zijn ‘minFiles’, ‘maxFiles’, ‘validFileExtensions’, ‘uploadFileExtensions’.

                  • ‘externalPageForInput’ bevat een url naar een externe pagina die input-bestanden stuurt.

                  output

                  Beschrijft de output van de workflow. Laat je deze eigenschap weg, dan is er op de workflow-page geen mogelijkheid om bestanden te downloaden. Onderliggende eigenschap is ‘type’. Deze kan de waarden ‘none’, ‘zip’ en ‘files’ bevatten.

                  private

                  De definitie van de volledige workflow is vereist. Deze blijft volledig server-side. Het bestaat uit twee onderdelen: ‘workflowVariables’ en ‘steps’. Deze worden in afzonderlijke tabellen behandeld.

                  2.1Onderdeel ‘input’

                  De meeste workflows zullen bestaan uit een invoerbestand, een transformatie, een uitvoerbestand. Omdat dit niet voor alle workflows hoeft te gelden is dit onderdeel optioneel.

                  Tabel 2Eigenschappen en onderdelen van input

                  Eigenschap, onderdeel

                  Omschrijving

                  files

                  De volgende vereisten voor de input-bestanden worden gecontroleerd:

                  • minimaal aantal bestanden,

                  • maximaal aantal bestanden,

                  • maximale bestandsgrootte van individuele bestanden,

                  • vereiste bestandsnamen,

                  • toegestane bestandextensies.

                  externalPageForInput

                  Deze eigenschap legt de url vast van een externe pagina, die invoerbestanden stuurt. In het onderdeel ‘Input’ van de workflow komt een knop om de pagina als popup te openen. Dit vervangt de dialoog voor het selecteren van een invoerbestand.

                  parameters

                  Deze optie is (nog) niet geïmplementeerd. Hiermee kunnen web-formulieren gegevens in velden doorgeven als workflow-variabelen.

                  Voorbeeld

                  …

                  "input": {

                  "files": {

                  "maxFiles": 20,

                  "uploadFileExtensions": [".xml", ".zip"],

                  "validFileExtensions": [".xml"],

                  "requireFilenames": ["opdracht.xml"]

                  }

                  },

                  …

                  2.2Onderdeel ‘output’

                  Veelal zal een workflow de uitvoerbestanden plaatsen in een repository of op een server. Het is vaak wel handig om de uitvoebestanden te downloaden voor controle. Daarom is er een optionele eigenschap ‘output’.

                  Tabel 3Eigenschappen en onderdelen van output

                  Eigenschap, onderdeel

                  Omschrijving

                  type

                  De volgende typen zijn mogelijk:

                  • ‘none’ toont geen output in de website.

                  • ‘zip’ toont een download-link waarmee de gebruiker de uitvoerbestanden als zip-bestand kan downloaden.

                  • ‘files’ toont download-links waarmee de gebruiker de afzonderlijk uitvoerbestanden kan downloaden.

                  Voorbeeld

                  …

                  "output": {

                  "type": "zip"

                  },

                  …

                  2.3Onderdeel ‘private’

                  De daadwerkelijke definitie van de workflow gebeurt in onderdeel ‘private’.

                  2.3.1Opmerkingen

                  Je kunt eigen opmerkingen toevoegen om de workflow begrijpbaar te maken. Hiervoor is het optionele onderdeel ‘comments’.

                  Voorbeeld

                  "private": {

                  "comments": [

                  "Runs the DSO_overzicht ANT script.",

                  "Step 1: Test if ANT is available.",

                  "Step 2: Run the DSO_overzicht ANT buildfile.",

                  "I had to adjust the xsl called by build.xml of DSO_overzicht.",

                  "the value passed to "-Dopdracht" was not capable of handling windows pathnames",

                  "The overzicht.xsl script can"t handle absolute paths outside REPO_DIR."

                  ],

                  …

                  2.3.2Variabelen

                  Binnen het beheerportaal kennen we ‘omgevingsvariabelen’ en ‘workflow-variabelen’:

                  • Voor de hele omgeving staan omgevingsvariabelen in het .env-bestand.

                  • In een workflow definieer je workflow-variabelen die in alle stappen gebruikt kunnen worden.

                  • In een stap kun je zowel omgevingsvariabelen als workflow-variabelen definiëren. Omgevingsvariabelen zijn alleen beschikbaar binnen de stap, workflow-variabelen ook binnen andere stappen in de workflow.

                  De schrijfwijze om een variabele aan te roepen is bijvoorbeeld ${REPO_DIR}. Bij het starten van een workflow worden altijd de volgende variabelen aangemaakt:

                  • ROOT_DIR is de root-folder die aangemaakt wordt voor de workflow.

                  • INPUT_DIR is de folder ${ROOT_DIR}/input. Deze folder bevat de bronbestanden.

                  • OUTPUT_DIR is de folder ${ROOT_DIR}/output. Deze folder bevat de resultaatbestanden.

                  • TEMP_DIR is een folder voor tijdelijke bestanden. Aan het einde van de workflow wordt deze leeggegooid.

                  Tabel 4Eigenschappen en onderdelen van workflowVariables

                  Eigenschap, onderdeel

                  Omschrijving

                  key

                  Unieke naam van de variabele. Deze mag alleen a-z, A-Z, 0-9 en _ bevatten.

                  value

                  Dit kan een string zijn, maar voor het werken met paden zijn er onderliggende eigenschappen:

                  • ‘type’ geeft het type waarde aan. Optie ‘path’ is gedefineerd, default is ‘undefined’.

                  • ‘path’ geeft het pad aan als string.

                  • ‘separator’ geeft het scheidingsteken aan als string. Default is ‘undefined’.

                  • ‘makeAbsolute’ geeft aan of de waarde als absoluut of relatief pad doorgegeven moet worden. Default is ‘false’.

                  hide

                  Boolean-waarde om aan te geven of de waarde in de log-bestanden zichtbaar mag worden.

                  Voorbeeld

                  "private": {

                  "workflowVariables": [

                  {

                  "key": "REPO_OWNER",

                  "val": "Geonovum"

                  },

                  {

                  "key": "ANT_BUILDFILE",

                  "val": {

                  "type": "path",

                  "path": "${REPO_DIR}/word2werkversie/build.xml"

                  }

                  },

                  {

                  "key": "INPUT_ABS",

                  "val": {

                  "type": "path",

                  "path": "${INPUT_DIR}",

                  "separator": "/",

                  "makeAbsolute": true

                  }

                  },

                  …

                  2.3.3Stappen

                  Er zijn twee typen stappen:

                  • ‘cmd’: de stap bevat een command-line opdracht.

                  • ‘xpath’: de stap bevat een xpath-expressie om een variabele op te halen uit een xml-bestand.

                  Beide bovenstaande typen stappen worden binnen een workflow gedefinieerd. Het is ook mogelijk om ze in een afzonderlijk bestand te definiëren en vanuit de workflow hiernaar te koppelen. Dit is vooral handig bij veel voorkomende stappen. Er zijn in totaal vier type-aanduidingen: ‘cmd’, ‘xpath’, ‘linked-cmd’, ‘linked-xpath’. We nemen per type de eigenschappen door.

                  2.3.3.1Command-line stap

                  Meest gebruikt zal de command-line stap zijn. Deze voert een command-line uit zoals je gebruikelijk in een shell zou doen.

                  Tabel 5Eigenschappen en onderdelen van step ‘cmd’

                  Eigenschap, onderdeel

                  Omschrijving

                  type

                  Type-aanduiding, in dit geval ‘cmd’.

                  name

                  Een leesbare naam van de stap voor foutmeldingen. Als dit niet ingevuld is, wordt automatisch een logische naam gekozen.

                  description

                  Een beschrijving van wat de stap doet.

                  command

                  De command-line opdracht, bijvoorbeeld ‘echo Hello World!’.

                  cwd

                  De Current Working Directory van waaruit de command-line opdracht gestart wordt. Default is ‘${ROOT_DIR}’.

                  envVars

                  Omgevingsvariabelen die alleen binnen de stap gebruikt kunnen worden.

                  workflowVariables

                  Workflow-variabelen die binnen alle stappen van de workflow gebruikt kunnen worden.

                  continueOnError

                  Een boolean-waarde om aan te geven of de workflow moet afbreken of doorgaan.

                  Voorbeeld

                  "steps": [

                  …

                  {

                  "type": "cmd",

                  "name": "git init",

                  "cwd": "${GIT_DIR}",

                  "command": "git init -b main"

                  },

                  {

                  "type": "cmd",

                  "name": "git add",

                  "cwd": "${GIT_DIR}",

                  "command": "git add --all"

                  },

                  {

                  "type": "cmd",

                  "name": "git commit",

                  "cwd": "${GIT_DIR}",

                  "command": "git commit -m "automated commit""

                  },

                  …

                  ]

                  2.3.3.2Xpath stap

                  In een workflow worden vaak gegevens gebruikt die in een eerste stap door een transformatie worden gegenereerd. Dit kan bijvoorbeeld door een config.xml mee te geven. Met de xpath stap kun je gegevens uit een xml-bestand lezen en in een workflow-variabele zetten.

                  Tabel 6Eigenschappen en onderdelen van step ‘xpath’

                  Eigenschap, onderdeel

                  Omschrijving

                  type

                  Type-aanduiding, in dit geval ‘xpath’.

                  name

                  Een leesbare naam van de stap voor foutmeldingen. Als dit niet ingevuld is, wordt automatisch een logische naam gekozen.

                  description

                  Een beschrijving van wat de stap doet.

                  xpath

                  De command-line opdracht, bijvoorbeeld ‘echo Hello World!’.

                  inputFile

                  Pad naar het json- of xml-bestand waar de xpath-query op uitgevoerd wordt.

                  toVariable

                  Naam van de variabele waarin het resultaat geplaatst wordt. Deze is daarna in de command-line van vervolgstappen te gebruiken.

                  workflowVariables

                  Workflow-variabelen die binnen alle stappen van de workflow gebruikt kunnen worden.

                  continueOnError

                  Een boolean-waarde om aan te geven of de workflow moet afbreken of doorgaan.

                  Voorbeeld

                  "steps": [

                  {

                  "type": "xpath",

                  "xpath": "config/item[2]/text()",

                  "inputFile": "input/config.xml",

                  "toVariable": "GET_ITEM"

                  },

                  {

                  "type": "cmd",

                  "command": "echo \"Xpath result: ${GET_ITEM}\""

                  }

                  ]

                  2.3.3.3Gekoppelde stap

                  De twee typen gekoppelde stappen zijn ‘linked-cmd’ en ‘linked-xpath’. Er zijn nu een aantal stappen gedefinieerd die vaak gebruikt worden: ‘ant-version.cmd-step.json’, ‘ant-buildfile.cmd-step.json’ en ‘xpath-query.xpath-step.json’.

                  Tabel 7Eigenschappen en onderdelen van step ‘linked-cmd’ en ‘linked-xpath’

                  Eigenschap, onderdeel

                  Omschrijving

                  type

                  Type-aanduiding, in dit geval ‘linked-cmd’ of ‘linked-xpath’.

                  linkedFilename

                  Bestandsnaam van de gekoppelde stap, bijvoorbeeld ‘xpath-query.xpath-step.json’.

                  overrideValues

                  De opgegeven overrideValues vervangen properties van de gekoppelde stap.

                  Voorbeeld

                  "steps": [

                  {

                  "type": "linked-cmd",

                  "linkedFilename": "ant-version.cmd-step.json"

                  },

                  {

                  "type": "linked-cmd",

                  "linkedFilename": "ant-buildfile.cmd-step.json"

                  },

                  {

                  "type": "linked-xpath",

                  "linkedFilename": "xpath-query.xpath-step.json",

                  "overrideValues": {

                  "xpath": "//github/repository/text()",

                  "toVariable": "REPO_NAME"

                  }

                  },

                  …

                  ]

                  3Werking van de API

                  Voor de werking van de website maakt het beheerportaal gebruik van een Postgres-database en een API die de communicatie tussen frontend en backend regelt. Hiervoor zijn nu de volgende endpoints in werking:

                  • Voor het registreren en inloggen van gebruikers,

                  • Voor het ophalen van pagina’s,

                  • Voor het uploaden van input-bestanden,

                  • Voor het uitvoeren van workflows en workflow-stappen,

                  • Voor het terugkoppelen van log-informatie,

                  • Voor het downloaden van output-bestanden.

                  Het inloggen en registreren van gebruikers wordt nu nog met een endpoint en registratie in de database gedaan, maar wordt vervangen door een Azure inlog-procedure.
